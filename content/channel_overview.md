## Content Pillars (Update Monthly)

YouTube punishes fuzzy channels. You need 3-4 clear pillars that repeat and form a pattern.

### **Pillar A: Creative Web / Creative Coding** (35% of content)
**Goal**: The person showing how to build visually interesting stuff for the web.

**Video Types**:
- TresJS/Three.js scenes (landing pages, scrollytelling, data-viz that doesn't suck)
- Shaders explained from web-dev angle ("GLSL for JS devs")
- Creative micro-projects: "Weird interactive cursor experiments", "3D transitions for portfolios"
- Visual experiments that make people say "I didn't know you could do that"

**Framework Rule**: Use Vue/Nuxt often, but occasionally framework-agnostic or small React/vanilla examples when it makes sense. Sell ideas, not frameworks.

### **Pillar B: OSS & DX (with AI in the mix)** (30% of content)
**Goal**: Be the reference for "how real tools get built".

**Video Types**:
- "Building X as an open-source library" (small utilities, TresJS add-ons, Vite plugins)
- DX walkthroughs: monorepos, release workflows, CLI tooling, tests, CI
- "AI as a maintainer assistant":
  - Using AI to triage issues, refactor, generate docs, write tests
  - Honest demos: where it helps, where it screws up

- Bruno Simon: no OSS focus

### **Pillar C: AI for Developers (practical, not hype)** (25% of content)
**Goal**: Own the practical intersection of AI and web dev.

**Video Types**:
- "AI pair-programming a creative feature" – building something real, fixing AI's mistakes on camera
- "Turn this boring CRUD into something smarter with AI" (summarization, smart search, etc.)
- "My actual AI dev workflow for day-to-day coding" – editors, prompts, guardrails
- AI for maintainers: "I used Claude to refactor TresJS core"

**Crucial Rule**: Every AI video must be anchored in code and outcomes, not opinions alone. Show the failures.

**Examples of Winners**:
- [Track your best performing AI videos here]
- What worked: [e.g., "Showed AI making mistakes and how I fixed them"]
- What flopped: [e.g., "Too much talking about AI, not enough using it"]

---

### **Pillar D: Dev Journey / Mentorship (tied to concrete work)** (10% of content)
**Why limited?**: Pure motivational content underperforms. Journey content needs code context.

**Rules**:
- ❌ NO: "You're not behind" standalone talk
- ✅ YES: "What I wish I knew before my first OSS project (while we build a tiny lib together)"
- ✅ YES: "Burnout lessons from maintaining a popular library" but with code context and actionable tips

**Every journey video must contain**:
- Actual code being written
- Tools or process being demonstrated
- Actionable takeaways, not just feelings
